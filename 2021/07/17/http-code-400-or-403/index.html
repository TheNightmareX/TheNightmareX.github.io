返回400还是403？

发表于 2021-07-17

根据规范，可以理解为：

400 - Bad Request 表示数据有误，如：

  • 请求体不是规范的 json 格式导致数据解析失败
  • 数据不符合要求
    • 缺字段
    • 字段类型有误
    • 字段突破限制（字符串过长、数字过大）

403 - Forbidden 表示权限不足，如：

  • 员工发送请求尝试删除老板
  • 员工尝试给自己加工资

通常认为返回403意味着这个用户以当前状态永远都无法访问这个接口（路径+方法），如某员工只要还是员工，就永远无法DELETE /people/boss/或是PATCH /people/myself/ { "salary": 999999 }，对于带有请求体的方法POST、PATCH、PUT，不论请求体什么样，该员工都会收到403的返回值

然而，对于POST /applications/ { "chatroom": <id> }接口，如果该用户先前已经创建了一个加入该聊天室的申请，且该申请还未被处理，则应拒绝该请求，在这种情况下，应该返回400还是403呢？

问了一圈儿，小白们似乎都认为应该返回403，因为该用户当前没有权限向该聊天室发送申请。

但是，还记得上面说的吗？

通常认为返回403意味着这个用户以当前状态永远都无法访问这个接口，如某员工只要还是员工，就永远无法DELETE /people/boss/或是PATCH /people/myself/ { "salary": 999999 }，对于带有请求体的方法POST、PATCH、PUT，不论请求体什么样，该员工都会收到403的返回值

该用户无法POST /applications/ { "chatroom": 1 }，但他可以POST /applications/ { "chatroom": 2 }，这就违反了直觉。故可以换一种角度，请求被拒绝的原因可以是请求体中的chatroom字段有误，用户不应填写一个已经存在未处理请求的chatroom，那么自然而然应该返回400了。
