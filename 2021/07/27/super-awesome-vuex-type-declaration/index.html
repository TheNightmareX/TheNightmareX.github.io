史上最完美的 Vuex 4 类型声明

发表于 2021-07-27 更新于 2021-08-01

前两天终于初步完成了后端。来到久违的前端，突然想起了 Vuex 恐怖的 TypeScript AnyScript 支持：我是绝不可能与这一大坨 any 同流合污的>_<！！然而费劲功夫却只找到寥寥几个解决方案：

  • TypeScript 4.1 新特性：字符串模板类型，Vuex 终于有救了？
  • 使用 Typescript 加强 Vuex 使用体验
  • danielroe/typed-vuex: 🏦 A typed store accessor for vanilla Vuex.
  • mrcrowl/vuex-typex: Typescript builder for strongly-typed access to Vuex Store modules
  • ktsn/vuex-smart-module: Type safe Vuex module with powerful module features
  • ktsn/vuex-type-helper: Type level helper to ensure type safety in Vuex

它们大多是通过提供额外的 API 来绕过 Vuex 内置的 AnyScript 类型声明，并要求定义一大堆多余的 interface 并在每次调用时作为泛型参数传入，使用起来麻烦，还使代码混乱难读，我都不满意。

那怎么办？自己动手，丰衣足食，既然 Vuex 的类型不够详细，那就用 declare module 补充声明不就可以了吗？ 咱也搞不懂这么简单的一个补充声明就能解决的问题为什么那么多人在用如此曲折的方法解决

环境

  • TypeScript - 4.3.5
  • Vuex - 4.0.2

特性

  • store.state 中包括自身与所有嵌套的子模块中的 state，如 store.state.module1.module2.value
  • store.getters 中包括自身与所有嵌套子模块中的 getters，如 store.getters['module1/module2computedValue']；且类型为对应的 getter 的返回值类型
  • store.commit 的第一个参数 type 会根据自身与所有嵌套子模块中的 mutations 进行自动补全，如 store.commit("module1/module2/increment", ...) ；第二个参数 payload 的类型与 type 所指向的 mutation 的第二个参数的类型相同
  • store.dispatch 基本同上，除了 store.commit 的返回值总是 void，而 store.dispatch 的返回值类型会与 type 所指向的 action 的返回值类型相同
  • 仅在 createStore() 的第二个参数 stricterTypes 为 true 时启用新类型

限制

只支持通过原始 store 对象执行操作（不支持任何 helper）。
